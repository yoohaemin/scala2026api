== Composing with flatMap

[%step]
[source,scala]
----
// Works with Either, ...
def checkDragonExists(id: DragonId)
  : IO[DragonNotFound, Unit] = ???

def checkDragonAvailable(id: DragonId)
  : IO[DragonVacationing, Unit] = ???
----

[%step]
[source,scala]
----
// Part of rentDragon
val check = for {
  _ <- checkDragonExists(id)
  _ <- checkDragonAvailable(id)
} yield ()
----

[%step]
[source,scala]
----
IO[DragonNotFound | DragonVacationing, Unit]
----

[%step]
Done?

[.notes]
--
Now, as Scala 3 users, we have a new feature compared to Scala 2: union types.

Let's say you have two operations: `checkDragonExists` and `checkDragonAvailable`, both returning `IO[..., Unit]`.

If you compose them with `flatMap`, you get that exact union type.

The advantage of this is that it is always precise, and it requires no manual maintenance.

// Slide 13 — C (code)
== Adding Stack Traces Manually

[%step]
[source,scala]
----
case class DragonNotFound(id: DragonId) {
  val stackTrace = ??? // 😩
}

case class DragonVacationing(id: DragonId) {
  val stackTrace = ??? // Again!!
}
----

[.notes]
--
So how do you get stack traces?

Simple: add the code that captures stacktraces into the constructor.
Repeat that in every case class you want to treat as an error.

But you don't want to repeat. How do you abstract over repeated code in the constructor?

That's right, you extend something. An abstract class or a trait.

== Widening

[%step]
[source,scala]
----
sealed trait Rejection extends RuntimeException

case class DragonNotFound(id: DragonId)
  extends Rejection
case class DragonVacationing(id: DragonId)
  extends Rejection
----

[%step]
[source,scala]
----

  _ <- checkDragonExists(id)
  _ <- checkDragonAvailable(id)
} yield ()
----

[%step]
[source,scala]
----
IO[Rejection, Unit]
----

[.notes]
--
But then you get this problem: the type inference becomes the supertype — in this case, `Rejection` — not `A | B`.

So you're thrown into a dilemma.

Either you keep vanilla case classes, or you accept that union type accumulation doesn't work.

So what does the backend app actually do?

It takes the shape implicitly requested by the frontend, and turns it into some mix of SQL, cache lookups, RPC calls, or stitching code.

That is where the interesting problem lives.

There are really two selection trees here.

One on the frontend-facing side of the backend: what shape does the REST endpoint return?

And one on the datasource-facing side of the backend: what shape do we fetch from below?

Same intent, duplicated around the backend.

// Slide 29 — S (statement)
== Selection-tree churn

[.notes]
--
This architecture works well when shapes are stable.

But in a fast-paced environment, they are not stable.

Frontend wants slightly different response shapes.

Product changes screens.

Design moves fields around.

And then the bottleneck is not raw runtime performance.

The bottleneck is engineering time spent updating those hardcoded selection trees.

// Slide 31 — S (statement)
== Backend selection trees are not

[%step]
* Hard to abstract query languages!
* Multiple sources!

[%step]
Local resolution

[.notes]
--
But representing selection trees inside the backend is definitely not a solved problem.

Why? Not through a lack of effort. We already had an excellent talk to abstract over SQL even today.

So much effort gone into abstracting over query languages. In my opinion, it's far from being conquered.

But it gets unwieldy once the shape becomes dynamic, or the query gets ugly.

If the data lives in multiple places, it just doesn't work.

My theory: there is no single mainstream declarative abstraction that spans all the kinds of datasources a backend talks to.
--

// Slide 32 — S (statement)
== Local resolution's first problem: N+1

[%step]
* Bad default: singular fetches
* Think: singular as specialized case of plural
* Compose singular & plural fetches in a single call

[.notes]
--
If we can't abstract selection trees, why don't we turn to local resolution?

First problem: N plus one.

And N plus one is not just a bug.

Repository modules tend to expose the minimum thing that solves the current task.

Then application code composes singular fetches in a loop, and you get N plus one almost by default.
--

Batching, deduplication, reasonable performance.

But look at this code. Do you see the selection tree?

// Slide 34 — S (statement)
== Local resolution's second problem: imperative code

[.notes]
--
Haxl did not solve the representation problem.

You still write the imperative program first.

Each step models a call to a datasource.

Each subroutine is built ad hoc.

The selection tree is still implicit in the control flow.

So even if execution is okay, ergonomics are still bad.
--

We want to represent the selection tree itself.

So just express the selection tree as data, and translate it into the imperative program.

You saw the API.

It's basically: get the dragon from the rental by following an edge, then get the breed and the vacation schedule of the dragon in parallel by following two more edges.

// Slide 40 — S (statement)
== Stitching as the default, SQL as optimization

[.notes]
--
Now I'm absolutely not claiming that you should never write joins anymore.

Joins are more likely to be faster than manual stitching with `ZQuery` or `Fetch`.

What I am claiming, though, is that once local resolution becomes ergonomic, it can be the sane default.

That is the change decrel is aiming for.

If fast UI iteration pushes you toward GraphQL, decrel can sit underneath it too. GraphQL resolvers need to solve the same backend problem.

And if you are on a REST backend and the real bottleneck is selection-set churn inside the backend app, decrel gives you a much nicer local-resolution default than one SQL string per endpoint.
--

== Only possible in Scala

[%step]
* Deriving error sets
1. Configurable type inference (`transparent`)

[%step]
* Decrel
1. Right-associative operators
2. Term inference (implicit search)
3. Module-scoped typeclass instances (lexical scope orphaned instances)
4. Production-ready

[.notes]
--
And I mean this quite literally: decrel is possible in this exact form because Scala has all the things I need.

Right-associative operators, so the tree shape reads naturally in the code.

Term inference, so you don't rely on a runtime registry of data fetching implementations.

Module-scoped typeclass instances, so one relation can have multiple implementations with the same typeclass.

Last but not least, it must be practical for mainstream backend application development.

Other languages have parts of these features, so decrel can be replicated in spirit.

But if you want this exact shape and this level of ergonomics, Scala is the only one I know.
